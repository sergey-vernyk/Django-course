# Bug Tracker API + Logging

## Опис
Реалізувати REST API для простого **Bug Tracker** та підключити логування подій.

## 1. Загальні вимоги

- Django + Django REST Framework
- Окремий `application` для роботи з багами
- Обробка даних через серіалізатори
- Логування в консоль та файл
- Без сторонніх бібліотек для логування

**Підказка:**
- Для API використовувати стандартні DRF views (`APIView` або generic views)
- Серіалізатори — на базі `ModelSerializer`
- Логування налаштовується через `LOGGING` у `settings.py`

## 2. Модель `Bug`

Модель повинна містити поля:

- назва бага
- опис
- рівень критичності
- статус
- дата створення

**Підказка:**
- Використовуйте стандартні Django-поля та `choices` для статусу

## ️3. Серіалізатор

Створити серіалізатор для моделі `Bug`.

Він повинен:

- працювати з моделлю
- включати всі поля
- виконувати валідацію даних

Валідація:
- `severity` — значення від 1 до 5
- логічна перевірка статусів

**Підказка:**
- Використовуй `ModelSerializer`, методи `validate_<field>` та `validate`

## 4. API ендпоінти

Реалізувати ендпоінти для:

- отримання списку багів
- створення бага
- оновлення бага
- видалення бага

**Підказка:**
- Можна використати generic views для CRUD-операцій

## 5. Логування

### Налаштування

- Логування описати в `settings.py`
- Використати:
  - логування в консоль
  - логування у файл
- Для app з багами використати окремий логер

**Підказка:**
- Використовуйте стандартні `StreamHandler` та `FileHandler`

### Події для логування

- створення бага
- оновлення бага
- видалення бага
- помилки валідації
- спроба доступу до неіснуючого бага
- неочікувані помилки

**Підказка:**
- Використовуйте рівні `INFO`, `WARNING`, `ERROR`

## 6. Використання логера

- Логер повинен використовуватись у `views`
- Логи мають зʼявлятись при ключових подіях

**Підказка:**
- Отримуй логер через `logging.getLogger()`
