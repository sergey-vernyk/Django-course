# Фінальний проєкт з Django

Фінальний проєкт — це **повноцінний веб-додаток на Django**, який демонструє:
- розуміння архітектури Django
- вміння працювати з базою даних
- роботу з користувачами та доступами
- уміння писати читабельний і акуратний код

## 1. Моделі та база даних

- Мінімум **4–6 моделей**
- Обовʼязково використати:
  - `ForeignKey`
  - `ManyToManyField`
- У кожній моделі:
  - метод `__str__`
- Усвідомлене використання:
  - `unique`
  - `blank`
  - `null`
- Логічні звʼязки між моделями (не формальні)

## 2. Адмінка Django

- Усі моделі зареєстровані в `admin`
- Для кожної моделі використати `ModelAdmin`
- Обовʼязково:
  - `list_display`
  - `search_fields`
  - `list_filter`
- Адмінка повинна бути придатною для реального використання

## 3. Користувачі та аутентифікація

- Реалізувати:
  - login 
  - logout
  - registration 
- Дані в проєкті **привʼязані до користувача** (замовлення, товари, ігри тощо)
- Користувач може редагувати / видаляти **тільки свої обʼєкти**
- Використати:
  - `LoginRequiredMixin` або `@login_required`

## 4. Views та URLs

- Використати **class-based views (CBV)** або **FBV** - функціональні:
  - `ListView`
  - `DetailView`
  - `CreateView`
  - `UpdateView`
  - `DeleteView`
- Логічна структура URL:
  - `app_name`
  - іменовані маршрути (`name=`)

## 5. Форми та валідація

- Використати **ModelForm** мінімум для 1–2 моделей
- Реалізувати **кастомну валідацію**:
  - `clean_<field>()` або `clean()`
- Валідація повинна містити бізнес-логіку

## 6. Templates

- Використати:
  - `base.html`
  - `{% extends %}`
  - `{% block %}`
  - `{% include %}`
- Передача даних з view у шаблони через `context` параметр в функції `render`
- Використати:
  - `{% if %}`
  - `{% for %}`
  - `{{ variable }}`
- Дизайн не оцінюється, але сторінки мають бути читабельні

## 7. Повідомлення та обробка помилок (не обов'язково, але це гарна практика)

- Використати `django.contrib.messages`
- Повідомлення про:
  - створення
  - оновлення
  - видалення
- Кастомні шаблони для:
  - 404
  - 403

## 8. Логування

- Використати стандартний модуль `logging`
- Логувати ключові події:
  - loging / logout користувача
  - створення / оновлення / видалення обʼєктів
  - помилки рівня `ERROR`
- Використати рівні логів:
  - `INFO`
  - `WARNING`
  - `ERROR`

## 9. Сигнали Django (буде плюс)

- Використати **хоча б один сигнал**:
  - `post_save`
  - `pre_save`
  - `post_delete`
  - `pre_delete` 
- Сигнали повинні виконувати корисну дію:
  - логування
  - створення повʼязаних обʼєктів
  - інша бізнес-логіка
- Сигнали винести в окремий файл (`signals.py`)

## 10. Налаштування проєкту

- краще використати `.env` файл і зберігати конфіденційні змінні там (треба додатково встановити `python-dotenv`)
- `SECRET_KEY` **не зберігати в коді**
- `DEBUG` керується через змінні середовища
- Наявний файл `.env.example`
- Проєкт не складається з одного appʼа

## 13. Стиль коду та форматування (ОБОВʼЯЗКОВО)

Код у проєкті **повинен бути відформатований** та відповідати базовим правилам `PEP8`.

Обовʼязково:
- відступи — **4 пробіли** (**2 tab**)
- порожні рядки:
  - між import-блоками і кодом
  - між функціями та класами
- логічне групування імпортів:
  1. стандартна бібліотека
  2. сторонні бібліотеки
  3. локальні імпорти
- читабельні назви змінних, функцій, класів

### Форматування через IDE
- Якщо використовується **PyCharm**:
  - код повинен бути відформатований стандартним форматером IDE
  - перед здачею проєкту обовʼязково використати
    - `Ctrl + Alt + L` - для форматування коду
    - `Ctrl + Alt + O`- для оптимізації (групування) імпортів
    
P.S. Це стандартна комбінація в `PyCharm`, але може бути у вас налаштована якась інша.

- Якщо використовується інша IDE:
  - використовуйте `black`, `ruff` або аналогічний інструмент

> Працюючий, але не відформатований код — вважається недоліком проєкту.

## 14. Додаткові плюси (не обовʼязково)

- Розширене логування
- Додаткові тести

Фінальний проєкт оцінюється не тільки за принципом  
"працює/не працює", а й за:

- якістю архітектури
- читабельністю коду
- дотриманням стилю
- використанням інструментів розробника
