# Система оформлення замовлення з Celery

Створити Django-додаток, у якому авторизований користувач може створити замовлення, а частина бізнес-логіки буде виконуватись асинхронно через Celery.

## 1. Створити моделі

### Order

Модель `Order` повинна містити такі поля:
- `user` - зв'язок з користувачем (`ForeignKey` до стандартної моделі `User`).
- `status` - статус замовлення (`pending`, `confirmed`).
- `external_reference` — значення, отримане після відповіді від зовнішнього API.
- `created_at` - дата створення.

### OrderItem
Модель `OrderItem` повинна містити:

- `order`- зв'язок із замовленням (`ForeignKey`).
- `product_name` - назва товару.
- `quantity` - кількість товару.

## 2. Створити view для оформлення замовлення

### View повинна:
- створювати об'єкт `Order`.
- створювати хоча б один `OrderItem`.
- запускати `Celery` задачу для обробки замовлення.
- запускати `Celery` задачу для надсилання email користувачу.

>Добре підійде окрема функція типу: `create_order_view(request: HttpRequest) -> HttpResponse`

## 3. Створити HTML template

### У template має бути форма з полями:
- `product_name`
- `quantity`

>Назву template можна зробити: `order.html`.

HTML можна зробити простим - головне, щоб форма передавала дані у view.

## 4. Створити Celery задачу для обробки замовлення

### Задача повинна:
- отримати `order_id` як параметр.
- завантажити замовлення з бази.
- перевірити, чи замовлення вже не оброблене (запит на сторонній API як робили на уроці, просто симуляція довгої обробки).
- отримати всі товари замовлення.
- відправити список товарів у тестовий API.

>Для тестування можна використати:
>https://httpbin.org/delay/2

### Після успіху:

- змінити статус замовлення на `confirmed`.
- зберегти `external_reference`.

## 5. У задачі використати `retry`

### Retry повинен спрацьовувати тільки для:
- `requests.Timeout`
- `requests.ConnectionError`

>Краще використати:
>- `bind=True`
>- `self.retry(...)`

Retry краще робити через `try/except`, а не через `autoretry_for`, щоб було зрозуміло, яка саме помилка обробляється і можна додати логування помилки.

### Обов'язково додати: `acks_late=True`
Якщо `worker` завершиться аварійно, задача повернеться назад у чергу.

## 6. Додати захист від повторного виконання
### Перед обробкою перевіряти:

- якщо статус вже `confirmed`, задача повинна завершитися.

#### Важливо:
Celery задачі бажано робити ідемпотентними.

## 7. Створити окрему `Celery` задачу для email після створення замовлення

### Email задача повинна:
- отримати `order_id`.
- знайти користувача замовлення.
- відправити email про успішне створення замовлення.

>У листі бажано вказати:
>- username користувача
>- дату створення замовлення

### Retry для email: `SMTPException`

### Бажано використати `transaction.atomic()` для роботи з БД.

## Що краще не робити:
- використовувати `Exception` для `retry`.
- повторювати задачу для логічних помилок.
- робити retry для `DoesNotExist`, якщо об'єкт в БД не знайдено.

Не всі помилки потрібно повторювати. Retry має сенс тільки для тимчасових проблем.

## Головна ідея
Потрібно показати, що Celery використовується для:
- повільних операцій.
- зовнішніх API.
- email відправлення.
- задач, які не повинні блокувати web request.
